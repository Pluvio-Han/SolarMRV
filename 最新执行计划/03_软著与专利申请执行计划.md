# SolarMRV 软著与专利申请执行计划

> 配套文档：`01_BP_碳核算MRV调整版.md` + `02_代码与技术执行计划.md`
> 适用阶段：校级 → 省级 / 国家级冲刺
> 版本：v0.1（2026-05）

---

## 核心结论

**不需要从零开发新软件**。SolarMRV 现有代码量（Python + Solidity + Next.js 加起来轻松上万行）已经远远超过软著申请所需的最低门槛（前 30 页 + 后 30 页源代码），关键是**怎么把现有系统"切片"成 2–3 个独立软著作品分别申请**，并同步启动 1 份发明专利申请，让评审在答辩时看到"3 份软著证书 + 1 份发明专利受理通知书"的硬证据组合。

申请软著的核心收益：

- **成本极低**：通过学校知识产权办公室代办，200–500 元/件，大多数学校有补贴甚至全免
- **周期短**：1–3 个月下证，赶得上省赛 / 国赛答辩
- **评审硬信号**：软著证书是肉眼可见的官方背书，是大创类比赛"上位圈"项目的标配
- **可以一稿多用**：1 套代码按模块切分，可合规拿到多个软著证书

## 三份软著的拆分方案

### 软著 1：SolarMRV 分布式光伏减排量 MRV 核算系统 V1.0

**性质**：最稳的一份，完全不涉及区块链字眼，评审环节零阻力。

**覆盖代码范围**：

- `solar_monitor.py`（光伏设备数据采集）
- `auto_monitor.py`（自动监测守护进程）
- `mrv/` 整个目录（核算引擎，按代码计划第一档完成后纳入）
- `config.py`（系统配置）
- `data/` 数据层相关代码
- `telegram_notify.py`（告警通知）

**功能描述要点**：

- 光伏设备直接采集与多协议适配
- 自动周期监测与异常告警
- 按 CCER 方法学口径折算减排量
- 多格式报表输出（PDF / Excel / JSON）
- 历史数据查询与图表生成

### 软著 2：SolarMRV 联盟链可信存证与核查接口系统 V1.0

**性质**：体现技术深度的核心软著，但需严格规避金融化字眼。

**覆盖代码范围**：

- `chain_client.py` + `chain_relay_v2.py`（联盟链客户端）
- `wallet_manager.py`（密钥管理）
- `contracts/SolarDataStore.sol` + `contracts/EmissionRecord.sol`（在用合约）
- `api/verifier/` 整个目录（核查接口，按代码计划第一档完成后纳入）

**功能描述要点**：

- 基于 FISCO-BCOS 国产联盟链的设备数据可信存证
- SM2 国密签名与签名校验
- 第三方核查机构专用 RESTful 接口
- 多节点共识与多方协同访问
- 链上原始记录哈希与减排量记录元数据查询

**⚠️ 严格规避字眼（材料里绝不能出现）**：

- RWA / Real World Asset / 现实世界资产
- Token / 代币 / 通证
- Mint / 铸造 / 铸币
- DEX / 去中心化交易所 / 交易撮合
- Stable Coin / 稳定币
- 任何涉及"资产价值转移"、"金融化"、"流转"、"兑换"的表述

**⚠️ 不纳入提交代码范围**：

- `contracts/deprecated/SolarRWA.sol`
- `contracts/deprecated/SolarToken.sol`
- `contracts/deprecated/StableCoin.sol`

这三份历史合约已归档至 `contracts/deprecated/` 子目录，作为 GitHub 仓库的技术原型存档保留即可，**不要**作为软著提交版的源代码片段。

### 软著 3（推荐）：SolarMRV 多端可视化看板与运营管理系统 V1.0

**性质**：相对容易过审，作为"工程产出密度"的补充。

**覆盖代码范围**：

- `frontend/` 整个目录（Next.js + TailwindCSS 前端）
- `api/` 后端服务（除 `api/verifier/` 外的部分）
- `assets/` 中的关键展示资源

**功能描述要点**：

- 多端响应式可视化看板
- 业主端 / 核查机构端角色权限分离
- 实时功率与历史发电量图表
- 减排量明细与累计统计
- 系统运行状态监控

## 实操步骤（建议 2–3 周内全部启动）

### 第 1 步：定名与划边界（半天）

- 给每份软著定一个正式中文名称（含 V1.0 版本号）
- 明确每份软著包含哪些代码文件与目录
- 在本目录下新建 `软著申请清单.md`，记录每份软著的边界与文件清单
- 提前与学校知识产权办公室对接，了解学校代办流程与所需材料

### 第 2 步：撰写软件说明书（每份 5–8 页，每份约 1 天）

每份软件说明书需包含以下章节：

1. **软件名称、版本与开发者信息**
2. **软件功能简介**（200–500 字）
3. **技术架构图**（一张框图即可）
4. **主要功能模块详细说明**（每个模块 1 段文字 + 截图）
5. **运行环境要求**（操作系统、Python 版本、依赖库等）
6. **典型使用流程**（含 3–5 张操作截图）
7. **关键算法或核心逻辑说明**（重点章节，体现技术深度）
8. **测试结果摘要**（可附 8 小时耐久测试图）

**建议**：可以让 codex 起草初稿，团队成员负责截图、配图、最后润色。

### 第 3 步：整理源代码提交版（每份半天）

软著要求提交：

- **前 30 页源代码**：从主程序入口开始的前 1500 行左右（每页 50 行）
- **后 30 页源代码**：从主程序末尾倒数的 1500 行左右
- **页码连续**：从第 1 页到第 60 页
- **每页 50 行左右**：单行不超过 80 字符
- **删除大段空行与冗余注释**

**关键预处理**：

- 对软著 2 而言，必须在提交版本里**全局替换**所有变量名、注释、字符串中的 `RWA`、`token`、`mint`、`dex`、`swap`、`stable coin` 等字样，改为合规表达（如 `record`、`store`、`emit` 等）。**注意**：这是提交版本的处理，**不影响 GitHub 仓库本身的历史记录**。
- 建议写一个简单的 Python 脚本自动化完成代码合并、去空行、加页码、关键词替换四步处理。

### 第 4 步：通过学校知识产权办公室代办

- 准备材料：软件说明书 + 源代码 + 申请人身份证复印件 + 学校开具的证明
- 通过学校代办的优势：价格便宜（200–500 元/件甚至全免）、流程熟、可同步纳入学校科研统计
- 处理周期：1–3 个月下证

## 同步申请发明专利

### 推荐专利名称

**《一种基于国密 SM2 与联盟链的分布式光伏减排量可信核算方法及系统》**

### 关键时间节点

- **申请号 + 受理通知书**：1–2 周即可拿到
- **公开**：18 个月左右
- **下证**：1–3 年

**重点**：**受理通知书**就足以写进 BP 和答辩 PPT，与软著证书是同等级的硬证据。不需要等到下证。

### 专利申请准备材料

1. **专利说明书**（10–20 页）：背景技术、发明内容、附图说明、具体实施方式
2. **权利要求书**（1–3 页）：1 项独立权利要求 + 若干从属权利要求
3. **附图**（架构图、流程图、时序图等 3–5 张）
4. **说明书摘要**（300 字以内）

**建议**：通过学校知识产权办公室代办，专利申请有学校补贴的话个人成本接近 0。

### 专利核心创新点（撰写专利时重点突出）

1. **国密 SM2 边缘签名与联盟链存证结合**：解决境外密码学方案合规风险
2. **链上减排量记录的不可转让设计**：与传统 RWA 代币方案的本质区别
3. **多方协同的核查接口架构**：业主、运营、核查、监管四类角色的权限分离与协同
4. **方法学版本与排放因子版本绑定**：每一笔减排量记录可被独立复算与版本回溯
5. **CBAM 时序匹配算法**（可选）：绿电产生曲线与企业用电曲线的小时级匹配

## 风险与注意事项

**风险 1：软著 2 审查关注度更高**
联盟链 / 区块链相关软著的审查相对严格，必须确保提交版本中无金融化字眼。建议提交前请学校知识产权办公室或导师再过一遍。

**风险 2：专利申请的"新颖性"**
申请前不要在公开渠道（GitHub README、博客、答辩公开材料）过早披露专利核心技术方案，避免影响新颖性认定。建议先提交专利申请、拿到申请号后再公开。

**风险 3：学校代办周期**
学校知识产权办公室代办流程可能有内部排队周期，建议尽早对接（最好 5 月下旬就启动），避免影响 6 月中旬的正式提交节点。

**风险 4：成员署名顺序**
软著与专利的作者署名顺序需要团队提前协商一致，避免后期纠纷。建议第一作者轮换分配到不同软著上，让每位核心成员都有"第一作者"署名。
