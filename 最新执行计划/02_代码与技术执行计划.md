## 第一档：必做（省赛冲刺前必须完成）

> 这一档决定合规叙事和技术深度是否站得住，缺了直接掉档。
> 预计代码量 1000–2000 行，2–3 周可完成。

### 1. 智能合约层的合规切割

**目的**：彻底剥离金融化属性，让评审一眼看到合规边界。

**任务**：

- ✅ 已完成：三份早期合约（SolarRWA.sol / SolarToken.sol / StableCoin.sol）已加入 DEPRECATED 注释头并移至 `contracts/deprecated/` 子目录归档，配套 `contracts/deprecated/README.md` 已说明停用背景与时间点。
- 扩展 `SolarDataStore.sol`，新增字段：
  - `methodologyVersion`（CCER 方法学版本号）
  - `emissionFactor`（区域电网基准线排放因子）
  - `gridRegion`（电网区域代号）
  - `emissionReduction`（折算后的减排量 tCO2e）
- 新建 `contracts/EmissionRecord.sol`：减排量电子记录合约，**只读、不可转让、不可销毁**，结构上类似 Soulbound Token 但**不命名为 Token**，统一叫 `Record`。明确禁掉 `transfer`、`approve`、`mint`（外部可调用的版本）等所有价值转移接口。

**关键产出**：合规切割后的合约源代码 + 停用说明文档。

### 2. MRV 核算引擎（Module D）

**目的**：把链上的"发电增量"按 CCER 方法学正式折算为"减排量"，这是新叙事的核心技术抓手。

**任务**：

- 新建 `mrv/` 目录，包含以下文件：
  - `mrv/methodology.py`：CCER 分布式光伏并网发电方法学的核算公式实现（`ER_y = EG_y × EF_grid,y`）
  - `mrv/emission_factor.py`：区域电网基准线排放因子库，覆盖华北、华东、华中、东北、西北、南方、西南 7 个区域，数据来源于生态环境部最新公告
  - `mrv/calculator.py`：把每一笔链上发电增量按公式折算为减排量记录，附带计算时间、方法学版本、EF 版本、源数据链上哈希等元数据
  - `mrv/report.py`：MRV 报表生成器，支持 PDF / Excel / JSON 三种格式输出
  - `mrv/__init__.py`：模块入口
- 在 `tests/` 目录补齐 `tests/test_mrv_calculator.py`、`tests/test_emission_factor.py` 等核算单元测试。

**关键产出**：可被命令行或 API 调用、能输出标准 MRV 报表的核算引擎。

### 3. 第三方核查接口（Module E）

**目的**：让核查机构能直接调用接口验证数据真实性，是新商业模式的关键技术抓手。

**任务**：

- 在 `api/` 目录新建 `api/verifier/` 子模块，提供 RESTful 接口：
  - `GET /verify/device/{device_id}/records?from=&to=`：拉取指定设备时间段内的原始链上记录
  - `GET /verify/signature/{tx_hash}`：验证某笔上链数据的 SM2 签名有效性
  - `GET /verify/emission/{project_id}/summary`：项目级减排量汇总
  - `GET /verify/emission/{project_id}/detail`：项目级减排量明细列表
  - `POST /verify/audit-log`：核查机构查询行为本身需写入审计日志
- 配套：
  - API Key 管理与权限控制（不同核查机构 / 咨询机构使用独立 Key）
  - 调用限流（基于 IP 与 API Key 双维度）
  - 审计日志持久化（写入数据库与链上双留痕）

**关键产出**：可对外提供给核查机构使用的 API 服务 + 自动生成的 OpenAPI 文档。

## 第二档：强烈推荐（国赛冲刺前完成）

> 这一档决定项目能否在国赛层级和其他项目拉开差距。
> 预计代码量 1500–3000 行，1–2 个月可完成。

### 4. 多节点联盟链真实部署

**目的**：把"联盟链"从单节点 demo 升级为真正的多方共识网络，答辩 demo 才有说服力。

**任务**：

- 部署 4 个角色明确的节点：业主节点、运营方节点、核查机构节点、监管 / 学校节点。
- 每个节点跑在不同 IP（可用同学的几台机器或几个云实例）。
- 配置节点权限管理：业主与运营方有写权限，核查与监管节点仅读。
- 在答辩 demo 中展示"同一笔减排量记录在四个节点上同步落账"的现场画面。
- 在 `README.md` 与展示材料中加入节点拓扑图。

### 5. 国密合规链路全覆盖

**目的**：把 SM2 / SM3 / SM4 国密体系覆盖到全链路，强化国产合规叙事。

**任务**：

- 将 SM2 签名扩展到：API 接口签名（核查方调用 API 也需 SM2 签名）、合约方法调用签名、报表导出签名。
- 引入 SM3 哈希替换可能用到的 SHA-256（数据完整性校验、上链前哈希计算等环节）。
- 引入 SM4 对敏感字段做对称加密（如 API Key 存储、设备私钥本地存储等）。
- 出一份《国密合规自查报告.md》放在 `项目总结报告内容/`。

### 6. 时序数据库与数据持久化

**目的**：把"学生作业"升级为"产品级架构"，光伏数据是典型时序场景。

**任务**：

- 把 `data/solar_data_*.csv` 替换为 TimescaleDB 或 InfluxDB。
- 新建 `db/` 目录，管理 schema 定义、迁移脚本和数据库连接配置。
- 在 `solar_monitor.py` 和 `auto_monitor.py` 中接入数据库写入逻辑，保留 CSV 输出作为冗余备份。
- 提供历史数据快速查询能力（按时间范围、设备 ID、项目 ID 多维度查询）。

### 7. 前端增加核查机构专用页面

**目的**：在答辩 demo 时现场展示"核查机构视角"，是非常有冲击力的演示场景。

**任务**：

- 在 `frontend/` 中新增 `/verifier` 路由，专门展示：
  - 链上原始记录列表
  - SM2 签名校验状态（每条记录可点击验证）
  - 减排量明细与计算公式回放
  - 查询审计日志
- 与业主看板做角色权限分离，登录后根据角色路由到不同首页。

### 8. 自动化测试与 CI

**目的**：体现工程化成熟度，是评审打"硬分"的关键信号。

**任务**：

- 将现有 `tests/` 目录的脚本规范化为 pytest 用例。
- 关键模块单元测试覆盖率目标 60%+：MRV 核算引擎、签名校验、链上交互、API 接口。
- 加入 GitHub Actions：每次 push 自动跑测试 + 代码质量检查（black / ruff / mypy）。
- README 顶部挂上 `Tests Passing` 与 `Coverage` 徽章。

## 风险与依赖

- **硬件依赖**：第二档"多节点部署"需要至少 4 台机器（或云实例）。建议提前向学校或同学借调资源。
- **政策依赖**：第一档"MRV 核算引擎"中的方法学口径与排放因子，需要持续跟踪生态环境部最新公告，避免使用过期数据。
- **团队依赖**：第一档建议由 1–2 名核心开发主导，第二档可拆分给更多团队成员并行推进。
- **CBAM 依赖**：欧盟 CBAM 实施细则仍在持续完善，第三档 CBAM 报表的模板可能需要根据 2026 下半年最新指引调整。
