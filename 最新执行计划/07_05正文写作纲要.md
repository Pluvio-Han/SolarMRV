## 零、全局原则（每一节都必须遵守）

### 0.1 结题视角零风险原则

**项目周期 2026.05–2027.05，所有成果与承诺都会在 2027.05 结题验收时被对照检查。** 写作时必须遵守：

1. **不承诺外部合作**：不写"已对接 X 家咨询机构"、"已与 X 家核查机构签约"、"已与 X 家园区达成合作"等表述。所有外部合作均以"拟探索""若条件具备""作为延伸方向"等条件式语气表达。
2. **不承诺收入数字**：不写"第 1 年实现 X 万营收"、"完成首笔 X 万订单"等具体收入承诺。效益预测节用"具备…价值实现潜力"、"未来若推广可…"等条件句。
3. **不承诺客户数量**：不写"服务 X 家客户"，统一改为"项目面向 X 类场景设计"。
4. **不承诺园区 PoC**：所有 PoC 仅承诺到"校园内 PoC 原型试运行"。园区 / 企业作为延伸方向写。
5. **成果数量统一**：3 份软件著作权申请或受理通知 + 1 份发明专利申请或受理通知 + 1 份原型系统试运行报告 + 项目总结报告 + 调研报告 + 答辩 PPT。任何节提到成果均与此一致。

### （一）项目概况（800 字）

**核心论点**：

绿链未来（SolarMRV）是面向分布式光伏的 MRV 可信数据底座，聚焦底层发电数据"难证明、难追溯、难复核"的行业痛点，为 CCER 申报、园区碳管理、ESG 披露与 CBAM 绿电出口证明等场景提供可信数据支撑。

**段落结构建议**：

1. **第 1 段（行业背景，约 250 字）**：从双碳战略 + 全国碳市场扩容 + CCER 重启切入，说明分布式光伏正从环境效益资产升级为可签发碳资产。引用国家政策方向：双碳目标、数据要素 × 三年行动计划、可信数据空间发展行动计划。
2. **第 2 段（行业痛点，约 250 字）**：分布式光伏底层数据呈现"采集分散、格式不统一、缺乏可信留痕、跨主体核验难"等问题。MRV 阶段依赖人工台账与事后补录，导致核查成本高、签发周期长。点出"行业不缺数据，缺的是可信、连续、可追溯的数据"。
3. **第 3 段（项目定位，约 200 字）**：项目以国密 SM2 + FISCO-BCOS 联盟链 + 物联网采集为技术底座，构建从设备直采到第三方核查接口的端到端 MRV 数据底座。明确不开展金融化资产发行与流转、不撮合交易、不开展公众募资、不承诺收益。
4. **第 4 段（项目愿景，约 100 字）**：通过技术原型升级与校园 PoC 原型试运行，形成可面向 CCER 咨询机构、第三方核查机构、园区碳管理部门、企业 ESG 披露与 CBAM 绿电出口证明等场景设计的可信数据产品原型，服务双碳、数据要素与绿色金融融合的实践与教育价值。

**必含关键词**：分布式光伏、MRV、CCER、双碳、数据要素、国密 SM2、FISCO-BCOS、可信数据底座、ESG、CBAM。

**绝对禁止**：RWA、代币、DEX、铸币、稳定币。

**对接点**：与 06 简介第 1 段、BP 第一章 1.1–1.6 保持一致。

### （三）生产或运营（1000 字）

**核心论点**：项目以 6 大模块构成端到端 MRV 数据底座；3 个核心模块已完成原型，3 个新模块在项目周期内重点补齐；所有模块组合形成可对外交付的可信数据产品形态。

**段落结构建议**：每个模块约 150–180 字，按"功能定位 → 技术实现 → 当前状态 → 与 MRV 链路的衔接"四段式描述。

**模块清单**：

1. **模块 A：光伏边缘采集网关（已完成原型）**
   - 基于 RS485 / Modbus 协议直采光伏控制器数据
   - 已实现真实数据与模拟数据双轨切换
   - 已通过 8 小时连续耐久测试，96 个采样点零错包、零丢失

2. **模块 B：国密 SM2 边缘签名层（已完成原型）**
   - 所有出厂数据在边缘端完成 SM2 数字签名
   - 私钥本地生成、不外泄
   - 满足《商用密码管理条例》合规要求

3. **模块 C：FISCO-BCOS 联盟链存证层（核心已完成，节点扩展拟推进）**
   - 已实现单节点链上数据写入与查询
   - 项目周期内拟扩展为 4 节点联盟链（业主 / 运营 / 核查 / 监管角色分离）
   - 数据存证合约采用纯数据合约设计，不涉及任何价值转移逻辑

4. **模块 D：MRV 减排量核算引擎（拟建设）**
   - 按 CCER 分布式光伏并网发电方法学，将链上发电增量按区域电网基准线排放因子折算为减排量
   - 输出含方法学版本与排放因子版本元数据的减排量记录
   - 支持 PDF / Excel / JSON 三种报表格式

5. **模块 E：第三方核查接口（拟建设）**
   - 提供专用 RESTful API 供核查方调用
   - 含 API Key 管理、调用限流、审计日志等机制
   - 所有查询行为本身写入审计日志，形成核查留痕

6. **模块 F：可视化看板与运营端（基础版已完成，核查端拟建设）**
   - 基于 Next.js + TailwindCSS 的响应式 Web 看板
   - 业主端已上线（pluviohan.com）
   - 项目周期内拟新增核查机构专用页面，与业主端做角色权限分离

**收尾段（约 100 字）**：六大模块组合形成"设备直采—国密签名—联盟链存证—减排量核算—第三方核查接口"五步可信链路，是项目对外交付的最小完整产品形态。

**必含关键词**：所有六大模块名称、五步链路、国密 SM2、FISCO-BCOS、MRV、核查接口。

**绝对禁止**：
- SolarRWA / SolarToken / StableCoin 这三个旧合约名（这三个合约已停用，正文里不提）
- 代币 / 铸造 / 兑换 / 交易撮合 / DEX 等所有金融化字眼

### （六）风险预测及应对措施（700 字）

**核心论点**：项目识别政策合规、技术稳定性、试点推进、数据安全、团队连续性五类风险，对应建立可执行的应对措施。

**段落结构建议**：每类风险约 130–150 字，按"风险描述 → 应对措施"两段式表述。

1. **政策合规风险**：
   - 风险：碳市场政策、CCER 方法学、CBAM 实施细则、"数据要素×"三年行动计划等仍在持续完善
   - 应对：团队建立政策资料库与持续跟踪机制（覆盖双碳、CCER、数据要素×、可信数据空间、绿色金融、监管口径六条主线）；项目严守合规边界，链上记录仅作为发电数据与减排量核算的可信留痕，不开展任何金融化资产发行与流转；所有链上记录不可转让、不可流通；早期金融化合约已停用并形成内部留痕。

2. **技术稳定性风险**：
   - 风险：硬件长期运行可能掉线、链上节点可能故障
   - 应对：已实现 Systemd 守护进程自启、Telegram 异常告警；项目周期内拟引入边缘端断点续传、链上数据自动校验与多渠道告警机制。

3. **试点推进风险**：
   - 风险：外部园区合作存在不确定性
   - 应对：项目仅承诺校园内 PoC 原型试运行（依托校内分布式光伏项目即可完成），园区合作仅作为延伸方向探索，不作为结题成果承诺。

4. **数据安全风险**：
   - 风险：设备数据涉及发电主体的运行信息，存在商业敏感性
   - 应对：所有数据上链前进行脱敏处理；联盟链采用许可制接入；建立 API Key 权限分级与审计日志机制；与潜在合作主体签订数据使用边界协议。

### （七）项目特色（700 字）

**核心论点**：项目具备真实硬件锚点、国密 + 国产联盟链合规栈、MRV 业务落点、主动合规转向、专创融合教育价值五大特色。

**必含关键词**：8 小时耐久测试、国密 SM2、FISCO-BCOS、CCER、CBAM、专创融合、合规自觉性、政策响应能力。
