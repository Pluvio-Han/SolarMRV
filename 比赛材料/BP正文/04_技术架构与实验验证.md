# 第四章 技术架构与实验验证

## 4.1 系统总体技术架构

`SolarGuard` 的技术架构围绕分布式光伏数据“从源头产生到可信使用”的完整生命周期展开，整体可划分为五个层次：设备采集层、边缘处理层、可信存证层、服务接口层和应用展示层。

在设备采集层，系统直接面向光伏控制器等物理设备，通过通信协议读取发电功率、电压、电流、电池状态和累计发电量等关键指标。这一层负责解决数据“能不能从真实世界采集上来”的问题，是整个系统的物理基础。

在边缘处理层，系统对采集到的数据进行结构化处理、签名封装、异常值处理和周期性组织，保证原始数据从一开始就具备统一格式和可信处理逻辑。这一层负责解决数据“能不能被可信组织起来”的问题。

在可信存证层，系统借助联盟链完成关键数据的链上写入、链上查询和历史追溯。联盟链在本项目中承担的是可信留痕和协同验证的角色，而不是单纯作为展示性区块链模块存在。这一层负责解决数据“能不能形成多方可核验的记录”的问题。

在服务接口层，系统通过后端接口与业务封装，为上层应用提供可调用的数据入口。这使得采集与存证后的数据不再停留在底层脚本中，而能够通过标准化方式服务前端展示和后续应用扩展。

在应用展示层，系统通过看板、图表、状态面板等形式，向使用者展示实时数据、历史记录和链上状态，使整个系统既具备技术闭环，也具备展示和演示能力。

从整体技术逻辑看，`SolarGuard` 并不是“把几个技术模块堆在一起”，而是通过统一的目标导向，把物联网、密码学、联盟链和可视化系统组织成一套完整的数据基础设施原型。

## 4.2 关键技术路线

### 4.2.1 光伏设备数据采集

项目围绕分布式光伏控制器等设备开展了数据采集实现，重点获取与发电过程和电池状态相关的核心指标，包括光伏功率、光伏电压、光伏电流、电池电压、电池电量和累计发电量等。系统在设计时既考虑了真实设备接入，也保留了模拟模式，以便在硬件不在场的情况下继续进行开发、演示和逻辑验证。

这一技术路线的意义在于，项目不是从“数据已经存在”的假设出发，而是从真实采集这一最底层问题开始构建。只有真正打通设备数据链路，后续的签名、上链和可视化才有现实意义。

### 4.2.2 数据结构化与签名机制

为了让采集到的数据具备后续使用价值，项目在原始读取之后会对关键字段进行结构化封装，并引入签名机制对数据进行保护。系统当前采用签名方案对关键数据内容进行处理，使数据在进入存证和业务场景之前具备明确的数据边界和可信属性。

这一部分的设计重点在于避免系统沦为普通监测软件。单纯展示数据并不能支撑跨主体协作和后续核验，而带有可信处理逻辑的数据则更接近可被业务场景采信的数字凭据。

### 4.2.3 联盟链存证与链上查询

项目采用 FISCO-BCOS 联盟链作为可信记录底座，设计了链上写入、状态查询和历史记录读取等基础能力。通过链上存证，关键发电数据可以在后续被追溯、查询和交叉验证，提升系统整体的可信度和审计友好性。

与公链式的开放金融逻辑不同，联盟链更适合当前项目所面向的多方协同和可控记录场景，也更符合当前中国大陆政策环境下对可信数据基础设施的实际需求。

### 4.2.4 自动监测与告警逻辑

项目已实现自动监测程序，能够按照固定周期采集数据、生成图表、保存本地记录并触发通知逻辑。相比一次性测试脚本，自动监测机制使项目具备持续运行和长期观察能力，能够更接近真实业务中的运维与数据治理场景。

这一技术模块的价值在于，它将单点实验能力提升为“可连续运行的系统能力”，使项目从“技术 demo”进一步走向“可复用的原型系统”。

### 4.2.5 API 与前端可视化原型

项目同时开发了 API 与前端看板原型，用于承接底层数据并展示关键状态。这种设计一方面提高了项目的可演示性，另一方面也为未来对接其他业务系统预留了接口条件。前端模块当前部分数据仍处于 mock 阶段，但其结构已经清晰呈现了未来平台化的发展方向。

## 4.3 关键代码与系统模块实现情况

从代码实现层面看，项目目前已经具备较为完整的模块划分和工程骨架。

在设备采集侧，系统已完成核心采集程序，支持读取光伏关键参数并生成符合业务需要的数据对象。相关实现不仅面向真实读取，也保留了模拟路径，增强了系统的开发韧性和展示可持续性。

在自动运行侧，系统已实现自动监测、定时采样、CSV 落盘、图表生成和消息推送逻辑，使项目具备了连续运行的能力，并形成了可视化的实验数据输出。

在区块链交互侧，系统已经完成联盟链客户端、隧道连接、写入调用、读取查询和链状态汇总等基础能力，并围绕智能合约构建了数据上链和原型验证逻辑。

在合约侧，项目已完成数据存证合约和绿色资产映射原型合约的设计与实现。虽然围绕代币映射和价值流转的部分在当前政策环境下不再作为主要对外叙事，但从技术角度看，这些合约仍体现了项目在链上业务逻辑上的探索深度。

在服务和展示侧，项目已完成 API 原型、钱包原型、仪表盘页面和前端组件雏形，形成了较完整的展示链路。即使部分接口目前仍有 mock 成分，整体工程结构已经说明，项目并非停留在实验脚本集合，而是在朝产品化原型持续推进。

## 4.4 已完成的实验与验证成果

### 4.4.1 硬件读取与数据生成实验

项目围绕真实光伏设备通信链路完成了关键指标读取实验，验证了系统在硬件接入条件下的数据获取能力。同时，为应对硬件不在场的现实限制，系统还构建了模拟模式，使得核心业务逻辑可以在无设备环境下继续验证和演示。

这意味着项目在实验设计上兼顾了真实性与可持续性，既有真实设备锚点，也具备脱离设备继续推进开发的能力。

### 4.4.2 自动监测与持续运行实验

项目已围绕自动监测程序完成持续采样、数据保存、图表更新和告警推送相关实验。这一部分的价值在于，它证明系统不仅能读取一次数据，更能在一段时间内持续运行并生成连续结果，为后续“可信记录”和“历史分析”提供基础。

从项目已有图表和脚本可以看出，系统已具备将实时数据转化为连续监测结果的能力，这使其更接近真实使用场景。

### 4.4.3 联盟链读写与链上验证实验

项目已完成链上读写相关实验，能够将光伏数据上链，并支持链状态读取、记录数查询、最新记录调用等操作。这部分实验说明，项目不是停留在“打算上链”的层面，而是已经完成了真实的交互验证。

联盟链实验的完成，使项目的核心价值主张获得了技术支撑：分布式光伏数据不只是被采集和展示，而且可以进入可信记录和多方可查询的状态。

### 4.4.4 原型机制验证实验

在技术探索过程中，项目还围绕绿色资产映射、自动结算和价值流转逻辑开展了原型验证。这部分实验体现了项目团队对于新能源数据向更高层业务场景延伸的思考与尝试，说明项目不是仅仅停留在运维监测层面，而是主动探索了数据如何服务上层场景的问题。

需要强调的是，在当前政策环境下，这部分成果在比赛材料中应更多作为“技术探索和机制验证”的体现，而非直接作为当前落地业务方向进行表达。

### 4.4.5 前端可视化与交互原型验证

项目已完成前端看板和页面原型的搭建，能够展示实时功率、累计发电量、链上状态等关键内容。可视化原型的价值不只在于美观，更在于它证明项目已经具备从底层数据到用户可感知产品界面的完整转换能力。

这对于创新大赛类比赛尤为重要，因为评审通常不仅关注技术逻辑，也关注产品形态是否清晰、是否具备“可展示、可理解、可想象”的应用前景。

## 4.5 当前技术完成度评估

综合现有代码、实验和原型情况，`SolarGuard` 当前的技术完成度可以概括为：

第一，核心技术链路已经明确并完成打通。项目已经实现了设备采集、签名处理、链上写入、状态读取、自动监测和前端原型等关键模块，形成了较完整的系统骨架。

第二，项目目前处于“原型验证基本完成、产品化程度仍需提升”的阶段。一些后端接口、前端展示和业务服务逻辑仍存在 mock 或半实装状态，这意味着项目在展示和论证上已经足够完整，但距离正式应用产品仍有进一步深化空间。

第三，项目具备继续迭代的良好基础。由于代码结构清晰、模块边界明确、实验记录较完整，后续无论是继续推进真实接口接入、补充场景服务能力，还是进一步优化展示和部署方式，都具备较好的工程延展性。

从比赛视角看，这种完成度是有说服力的。它说明项目并非停留在“想法阶段”，而是已经完成了一套真实可展示、可验证、可继续深化的系统原型。

## 4.6 后续技术迭代方向

在恢复硬件条件和继续推进项目后，`SolarGuard` 的技术迭代可以沿着三个方向展开。

第一，进一步增强设备侧接入能力和多设备兼容性，使系统能够支持更多分布式光伏场景和更丰富的数据采集需求。

第二，完善 API、前端和业务服务层，降低 mock 成分，提高系统在真实应用场景中的可用性和可交互性，推动原型系统向更高完成度的产品形态演进。

第三，围绕绿证衔接、ESG 披露、绿色金融辅助尽调等场景，进一步设计更加贴近政策和市场需求的上层功能，强化系统从“可信数据原型”向“可信数据服务平台”的转化能力。

这些方向都建立在现有代码和实验基础之上，说明项目后续发展并非重新开始，而是在已有成果上持续迭代。

## 4.7 本章结论

总体来看，`SolarGuard` 已经完成了一条具有现实意义的技术验证路径：从分布式光伏设备数据采集出发，通过结构化处理、签名机制、联盟链存证、自动监测和前端可视化，形成了一套相对完整的可信数据服务原型。

这一结果说明，项目不仅在概念上成立，而且在代码、实验和系统实现层面具备真实支撑。虽然当前仍有部分模块需要进一步完善，但其核心技术框架、关键逻辑和实验基础已经基本成型，足以支撑项目进入更高质量的商业计划书撰写和创新创业赛事论证阶段。
